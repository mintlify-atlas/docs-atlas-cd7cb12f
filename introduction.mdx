## What is HTTPower?

HTTPower wraps your existing HTTP client (Finch, Req, or Tesla) with a production reliability layer. Instead of implementing circuit breakers, rate limiters, and retry logic yourself, HTTPower provides them as built-in, composable features that work across all supported adapters.

```elixir mix.exs
def deps do
  [
    {:httpower, "~> 0.20.0"},
    {:finch, ">= 0.19.0"}  # or :req or :tesla
  ]
end
```

## Why HTTPower?

Most HTTP client libraries leave reliability concerns to the caller. This means every team reimplements the same patterns — often incorrectly or inconsistently.

HTTPower solves this by providing battle-tested reliability patterns as first-class features:

- **Never raises exceptions** — every HTTP operation returns `{:ok, response}` or `{:error, reason}`
- **Adapter-agnostic** — switch between Finch, Req, and Tesla without changing your application code
- **Layered configuration** — set sensible defaults globally, override per-client, override per-request
- **Production profiles** — pre-tuned settings for payment processing, high-volume APIs, and microservices

## Quick example

```elixir
# Create a payment client with production-ready defaults
client = HTTPower.new(
  base_url: "https://api.stripe.com",
  profile: :payment_processing,
  headers: %{"Authorization" => "Bearer #{api_key}"}
)

# Make requests — circuit breaker, rate limiting, and dedup are automatic
case HTTPower.post(client, "/v1/charges", body: charge_params) do
  {:ok, %HTTPower.Response{status: 200, body: charge}} ->
    handle_success(charge)

  {:error, %HTTPower.Error{reason: :service_unavailable}} ->
    # Circuit is open — payment gateway is having issues
    use_fallback_processor()

  {:error, %HTTPower.Error{reason: :too_many_requests}} ->
    # Rate limit exceeded
    schedule_retry()
end
```
